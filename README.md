# Behavioral Threat Hunting

Behavioral Threat Hunting is a **self-hosted security monitoring and threat hunting platform** designed for detecting suspicious activity in web applications.

The system combines **behavioral detection, log analysis, metrics telemetry, and visualization** into a single platform built around modern observability tooling.

The platform integrates:

* **Sentinel Engine** — behavioral threat detection
* **Prometheus** — metrics collection
* **Grafana** — security dashboards
* **Loki + Promtail** — log aggregation
* **Portal API** — dashboard orchestration layer
* **Portal UI** — primary user interface
* **Homarr** — infrastructure service launcher
* **Demo Vulnerable Web App** — attack simulation target

---

# Platform Overview

The system monitors HTTP access logs generated by a vulnerable demo application.
These logs are analyzed by the Sentinel Engine which detects suspicious behavior patterns such as scanning activity or path traversal attempts.

Metrics and events are exported to observability tools and visualized through dashboards.

Users interact primarily with the **Portal UI**, while Grafana provides deeper analytics.

# Core Components

## Sentinel Engine

The detection engine responsible for identifying suspicious patterns in HTTP logs.

Examples of detections:

* Path traversal attempts
* Automated scanners
* Sensitive file access
* Suspicious query patterns

The engine exports metrics such as:

* processed log lines
* alerts per category
* alerts per IP
* targeted endpoints

---

## Prometheus

Prometheus collects metrics exposed by Sentinel Engine.

Examples:

```
sentinel_lines_processed_total
sentinel_alerts_total
sentinel_alerts_by_source_ip_total
sentinel_alerts_by_path_total
```

These metrics enable real-time analysis of attack patterns.

---

## Grafana

Grafana provides the main **security dashboards**.

The primary dashboard includes:

* Sentinel lines processed
* Alerts by category
* Alert rate over time
* Top attacking IPs
* Most targeted URLs

Grafana is also embedded into the platform workspace.

---

## Loki + Promtail

Loki stores application logs and enables log search within Grafana.

Promtail ships logs from the demo web server to Loki.

This allows investigators to view raw attack logs.

---

## Portal API

The Portal API acts as the **control plane** for the platform.

Responsibilities include:

* dashboard registry
* layout configuration
* service abstraction
* integration with Grafana dashboards

---

## Portal UI

The Portal UI is the **primary interface for analysts**.

Users land here instead of directly accessing Grafana.

Capabilities:

* curated security dashboards
* embedded analytics panels
* simplified interface for investigation

---

## Homarr

Homarr provides an **operations launcher dashboard** for the platform.

It aggregates links to:

* Portal UI
* Grafana
* Prometheus
* Loki
* Sentinel API
* Demo web application

This enables quick navigation across the stack.

---

## Demo Vulnerable Web App

The demo application simulates a vulnerable environment used for generating attack traffic.

A log generator sends malicious requests to simulate:

* scanners
* traversal attempts
* credential probes
* sensitive file access

These events feed the detection engine.

# Example Attack Simulation

Generate traffic manually:

```
curl http://localhost:8088/.env
curl http://localhost:8088/phpmyadmin
curl http://localhost:8088/wp-login.php
curl "http://localhost:8088/search?q=' OR 1=1 --"
curl "http://localhost:8088/admin?cmd=cat+/etc/passwd"
```

These requests will appear in:

* Sentinel alerts
* Prometheus metrics
* Grafana dashboards
