# Behavioral Threat Hunting

Behavioral Threat Hunting is a **self-hosted security monitoring and threat hunting platform** designed to detect suspicious activity in web applications through behavioral analysis.

The platform combines **behavioral detection, log aggregation, metrics telemetry, and observability dashboards** into a unified security analytics environment.

The system integrates the following components:

* **Sentinel Engine** — behavioral threat detection engine
* **Prometheus** — metrics collection and time-series database
* **Grafana** — security dashboards and visualization
* **Loki + Promtail** — log aggregation pipeline
* **Portal API** — dashboard orchestration and service registry
* **Portal UI** — analyst interface and investigation workspace
* **Homarr** — infrastructure launcher dashboard
* **Demo Vulnerable Web App** — simulated attack target

---

# Platform Overview

The platform monitors HTTP access logs generated by a vulnerable web application.

These logs are processed by the **Sentinel Engine**, which detects suspicious behavioral patterns such as:

* automated scanning
* path traversal attempts
* sensitive file access
* credential probing
* malicious query strings

Telemetry from the detection engine is exported into the observability stack and visualized using dashboards.

Security analysts interact primarily with the **Portal UI**, while **Grafana** provides deeper observability.

# Core Components

## Sentinel Engine

The Sentinel Engine is responsible for detecting suspicious activity in HTTP logs.

Example detections include:

* path traversal attempts
* automated scanners
* sensitive file access
* suspicious query patterns

The engine exports telemetry metrics such as:

```
sentinel_lines_processed_total
sentinel_alerts_total
sentinel_alerts_by_source_ip_total
sentinel_alerts_by_path_total
```

These metrics enable real-time attack analysis.

---

## Prometheus

Prometheus collects metrics emitted by the Sentinel Engine and other platform components.

Prometheus provides the time-series backend powering security dashboards.

---

## Grafana

Grafana provides the primary **security dashboards** for the platform.

Example visualizations include:

* Sentinel lines processed
* Alerts by category
* Alert rate over time
* Top attacking IPs
* Most targeted endpoints

Grafana dashboards are also embedded into the investigation workspace.

---

## Loki + Promtail

Loki stores application logs while Promtail ships logs from the demo web server.

This enables investigators to search raw attack logs directly from Grafana.

---

## Portal API

The Portal API acts as the **control plane** for the platform.

Responsibilities include:

* dashboard registry
* service discovery
* layout configuration
* Grafana dashboard integration

---

## Portal UI

Portal UI is the **primary analyst workspace**.

Users access dashboards and security insights through this interface rather than directly accessing Grafana.

Capabilities include:

* curated security dashboards
* embedded observability panels
* simplified threat investigation workflow

---

## Homarr

Homarr provides an **operations launcher dashboard**.

It aggregates links to platform services such as:

* Portal UI
* Grafana
* Prometheus
* Loki
* Sentinel Engine
* Demo web application

This provides a centralized entry point into the platform.

---

## Demo Vulnerable Web App

The demo application simulates a vulnerable environment used for generating attack traffic.

A traffic generator sends malicious requests to simulate:

* scanners
* traversal attempts
* credential probes
* sensitive file access

These events feed the detection pipeline.

# Running the Platform

The recommended way to run the platform is through **Docker Compose**.

### Start the platform

```bash
cd deployments/docker-compose
docker compose -f docker-compose.dev.yml up --build
```

### Services Started

```
demo-webapp
log-generator
sentinel-engine
postgres
redis
portal-api
portal-ui
loki
promtail
prometheus
grafana
homarr
```

# Development Setup (Manual)

If you want to run frontend services locally instead of Docker:

### Install dependencies

```
cd apps/portal-ui
npm install
```

### Run development server

```
npm run dev
```

Note:
`node_modules` is intentionally excluded from the repository and will be installed locally.

# Example Attack Simulation

Generate malicious traffic:

```
curl http://localhost:8088/.env
curl http://localhost:8088/phpmyadmin
curl http://localhost:8088/wp-login.php
curl "http://localhost:8088/search?q=' OR 1=1 --"
curl "http://localhost:8088/admin?cmd=cat+/etc/passwd"
```

These requests will appear in:

* Sentinel alerts
* Prometheus metrics
* Grafana dashboards
